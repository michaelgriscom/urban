1     Introduction [Back to Top]

This document outlines the testing procedure used in creating the URBAN software. The testing philosophy is outlined, along with test cases used.

2     Testing Philosophy [Back to Top]

We focus on black-box testing, using test programs in URBAN to simulate real usages. First, we test each assembly instruction (MOVD, IADD, etc) by itself, to ensure that the assembler accepts the right syntax and produces the right output. Then we perform integration testing against several logically related instructions and directives, making sure that they interoperate as expected. Finally, we maintain a few complete URBAN programs to test the entire system, both as functional programs to test the entire system, both as a system test and regression test when needed.

We try to use automated techniques such as JUnit to increase the ease (and therefore the likelihood and frequency) with which we run tests. Automation also reduces the likelihood of human error in checking the tests, although it does necessitate the need for caution when writing test cases.

3     Files Used in Testing [Back to Top]

3.1                      Syntax Test Files

The first type of test files are syntax check files, used for unit testing. These contain several logically independent lines. That is, except for a fewer header lines, each line is effectively its own test. They are meant only to detect invalid operands: an illogical operand, like LR with a MOVD instruction; an illegal value, like FM:5000; or a nonexistent keyword, like FOO:100; or the like. Lines without errors have no comments. Lines with errors should include a human-readable description of the error, to help a tester easily determine if the assembler reported the error properly. They should also include a list of numeric error codes, in the format "Errors: ### ### ###" (with the quotes), to assist in automating tests.

3.2                        Integration Test Files

These files are used to test the processing of directives and other functionality that are more complex than can be tested by a syntax test file alone, but that do not rise to the level of a complete program. Many directives, such as equates, need multiple lines to be completely tested, but still ought to be the focus of a test, not part of a bigger program. These programs may or may not contain intentional errors, depending on what is being tested. They will be manually run. Comments should supply a report of any expected errors and expected binary code.
