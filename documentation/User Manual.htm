3     Introduction [Back to Top]

This document outlines the process of installing and interacting with URBAN (Universal Robust Binary Assembler-nator) interpreter software. The hardware system is first described, followed by specifications for writing programs in the URBAN language. Assembler interaction is then detailed, along with the output from assembling sample source code. Error messages are detailed in the subsequent section, followed by a list of known software limitations.

4.2                      Urban Line Specifications [Back to Top]

The general instruction format is given below in Table 3. In the first column of a given line, a label is optionally given. The label may contain up to 32 alphanumeric characters beginning with an alphabetical character. If there is no label, the first column must be whitespace. The label is followed by a case-insensitive string representing the operation to be performed or a directive to the assembler/linker. Following this mnemonic, operands are given. The URBAN language uses keyword-driven operands, that is, the source and destination for each argument must be specified. These keywords are given in Table 3 of the Developers Guide and are also case-insensitive. Numeric arguments to keywords are always given in base 10. A mixture of regular expressions and more readable symbolic regular expressions will be used in this User Guide to define syntax. 4.3                      Directive Specifications [Back to Top]

Directives follow a similar format to operations, with a few differences because they are usually intended to be instructions to the assembler or linker rather than instructions for the program itself. Some of them have required labels, others require the lack of a label. Also, many of them do not impact the location counter, and can generate dynamic binary or even none at all. The Developers Guide details their symbolic formats, descriptions, and whether or not each impacts the location counter.

5     Program Interaction [Back to Top]

5.1                      Input Format [Back to Top]

The input to the assembler is expected to be strictly-compliant URBAN programs. Urban programs must be written in ASCII text with base 10 numbers (unless clearly specified as a different base) and must begin with a valid KICKO. Each line can contain only one URBAN opcode with its keywords, must end with a semicolon, and cannot have comments unless they are after the semicolon. On a line, the first character must either be a valid label name or whitespace if the line is unlabeled. After the optional label, a required opcode follows with proper keywords and values delimited by commas.

The assembler outputs an object file (described below) for each URBAN source file given to it. The linker reads in up to five object files concatenated onto each other and links these together into a module. Object files must be concatenated into one file for the linker to link them together into a module.

The linker produces a load file, which amounts to a simplified version of the assemblers object files, and this load file is executed by the simulator.

5.2                      Output Format [Back to Top]

The assembler output will be three-part: a source-listing, a symbol table, and an object file.

The linker will output its own symbol table and user report along with an executable load file.

The simulator will read in data from STDIN and print to STDOUT, and it has a built-in debug mode described below.

5.2.1        Assembler User Report

The User Report is the information printed to the screen during compilation. It prints three or more lines of output for each source line in the input file. The first line of each is simply an exact copy of the source line, this is followed by the line number within the source file, the lines location counter within the program, and the label for it (if it has one). Following this is the URBAN instruction or directive that has been parsed out of the line, and the binary equivalent for this instruction or directive. If any operands are present, they are printed on the next lines, along with their binary equivalents with one operand per line. If any errors were found within the line, a line containing the word Errors: is printed next, then each error is printed on an additional new line. Errors fall into three categories, as described in Section 14

5.3.5        Linker Symbol Table

The linker symbol table shows all symbols seen by the linker with their assembler assigned and linker assigned addresses. It also displays the length of the symbol in hex, typically one for most symbols other than program names. Assembler assigned addresses are respected when possible, that is, the first symbols linker assigned address (the first programs KICKO location) is guaranteed to be whatever the assembler assigned address was and other addresses are only changed if the memory they were assigned to is already occupied. Also, assembler assigned areas of empty memory are respected, so if the assembler specifies twenty bytes of empty memory between two programs, the linker places the start of the second program twenty bytes after the end of the first program.

5.3.7        Simulator Debug Mode

The simulators debug mode is activated with a command line flag of -d to make the simulator print all known information about every instruction before and after execution of the instruction. Included in the debugging information is the location counter, the contents of memory at the location pointed to by the location counter in both binary and hex, the opcode in binary, the register(s) referenced by the instruction, and the value of a literal constant in the instruction or contents of memory at that location if its a memory reference. 5.5                      Running the Assembler, Linker, and Simulator [Back to Top]

5.5.1        Starting the Assembler

Use java jar Urban.jar to start the assembler. The Browse button can be used to load a file, and then the Assemble button assembles it. The tabs below display the Urban Source, Symbol Table, and the Intermediate Version. When compiling from source, use javac Urban.java to compile, and then run the class file with java Urban. Be sure that the errors.txt and ValidOps.txt files are in the same directory as your class files if you run it this way, these files are required for the assembler to run correctly.

5.5.2        Load and Assembly

To load an URBAN file into the assembler, first click the Browse button to the right of the Input File box, then navigate to file wanted and click Open. After this, click the Assemble button to run the assembler, and view the output files in the tabs provided.

5.5.3        Linking _top

Linking can be performed from the command line given a load file by using java jar Linker.jar foo.obj output/directory/. Alternatively, the shortcut script Linker.sh (or Linker.bat if you are stuck using Windows) can be used to launch the linker, with the same arguments (object file and output directory). The linker user report, symbol table, and the load file will be written to the supplied directory as foo_report.out, foo_sym.out, and foo.out respectively.

To link all test files at once, the script SimpleLinker.sh (or SimpleLinker.bat) is useful. Simply run this from the directory it is in (the sp3 directory) and it will invoke the linker on all *.obj files in test-programs/linker/test/ and put the output in test-programs/linker/output. The output can be compared to test-programs/linker/expected to verify correctness (although the dates may not match).

5.5.4        Simulation

Simulation can be performed from the command line given a load file by using java jar Simulator.jar loaderFile.out 0. Appending a 1 flag instead of a zero after the load file argument will turn on debugging mode.
