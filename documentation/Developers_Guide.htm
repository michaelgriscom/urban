2     Introduction [Back to Top]

This document outlines the process of developing for the URBAN (Universal Robust Binary Assembler-nator) interpreter software.

3     Description of Hardware [Back to Top]

3.1                      Memory [Back to Top]

Memory consists of 4096 words, each containing 32 bits. Smaller units of a word (such as individual octets) are not directly addressable. Any of the 4096 words can be used for either data or program storage; that is, there is no separation between program and data memory.

3.2                      Registers [Back to Top]

There are 8 general-purpose registers. Each register is 32 bits. General-purpose registers are used for arithmetical and bit-level operations. There are also 7 index registers, also 32 bits. The index registers are used for Indexed addressing, described below.

3.3                      Addressing Modes [Back to Top]

Addressing modes can be thought of in three ways:

3.3.1        Register to Memory/Memory to Register

Data is transferred from a register to main memory, or from memory into a register.  Any of the eight general-purpose registers may be used as the destination or the source. Instructions using this addressing mode include the data movement instructions (such as MOVD) and the arithmetic instructions (such as IADD).  An instruction using Register to Memory/Memory to Register expects to receive either operands FR:REG,DM:MREF (for From Register:Register and Destination Memory:Memory Reference) or FM:MREF,DR:REG (for From Memory and Destination Register).  The addressing data is encoded in the binary using bits 7-13, except memory addresses, which are stored in 22-32.  The direction is set in bit 7.  If bit 7 is zero, the transfer goes from memory into the register, otherwise it goes from the register into memory.  Bits 8-10 address general-purpose registers, bits 11-13 indicate the index register (three zeroes here indicates no index register used).  Literals and constants can also take the place of a memory reference for some instructions; more information on them is below.

3.3.2        Indexed

Like either Register to Memory or Memory to Register, but with the addition of an index register. The value of the index register is added to the specified memory address, and that sum is used in the transfer. An instruction uses indexed addressing if a DX or FX keyword is present. This operand specifies which index register is used. The binary is the same as for Register to Memory/Memory to Register mode, but with bits 11-13 set to a value from 1 to 7 inclusive, specifying which index register to use.

3.3.3        One Register

Some instructions, such as CLR or IWSR require only one register. The register is specified by a DR:REG or DX:REG parameter (with the second being allowed only for CLR). The register is set in bits 8-10 of the binary if it is a general purpose register, and bits 11-13 for an index register. No registers Instructions such as CLRA, CLRX, HLT, DUMP do not require any registers or memory addresses to be specified.

4.2.1        Literals and constants

Constants and literals have close but slightly different usages.  Constants are used as parameters to mnemonics, but not generally as data for them to act upon, they are usually a value to be used but not updated during execution.  Literals are more frequently data that will be acted up or modified in some way.  Examples of common uses for constants include shift and rotated commands, where the constant defines the number of bits to be shifted/rotated, also DMP and HLT, where the constant determines the type of memory dump to execute and the value to return, respectively.  Examples of common uses for literals include comparison values (such as in a PST instruction) or substituting for a memory reference.  A literal can be substituted anytime a memory reference is expected because bit 19 is reserved for indicating whether or not the next 13 bits are a literal or a memory reference.

5     Assembler Implementation Overview

A GUI is used as a front end masking the internals of the assembler.  Inside, two passes are run over the URBAN source code:  one that does most of the work and catches most errors, and a second that finishes linking symbols with addresses and catches a few more errors. 

5.1                      First Pass [Back to Top]

The first pass is handled by the Parser class, which returns a Program class object.  This Program object contains all the data needed for the second pass to run, and the second pass never actually sees URBAN source.  The program object contains a Line object for each line in the file, and the lines contain everything specific to that line.  A Line object contains the raw URBAN source, the line’s label (if any), its directive or instruction, its address within the program’s memory, a list of operands, a set of errors, and the operands in binary.  These values are set at different points in pass one by the Parser, Line, InstrucTable, and Instruction classes.  The InstrucTable class handles validating mnemonics and checks ENT/EXT operands, while the Instruction class handles checking all other operands with calls to the methods on the Operand class.  The Program class adds all the lines to itself and builds a symbol table at the same time.

After all the values for a line are set, the main program calls the printListing() and printSymbols() methods on the Program object to have each line print all of its associated information and print the symbol table.  These are then forwarded to the GUI.

5.1                      Second Pass [Back to Top]

The second pass is called as a method on the Program.  For each line in the Program, the operands are evaluated and the binary for them is set.  Following this, the main program calls the Program object’s generateObjectFile()method, which calls five generateRecord() methods to generate each kind of record for the object file, as described in the User Manual.  First the header record is created, then generateObjectFile() iterates over each line in the Program to produce each line’s text record.  Each line is then iterated over again to produce modification records, which tell the linker how each line’s memory references should be modified.  Lastly, the end record is created, and these records are all concatenated together and sent to an output file and the GUI.

6     Linker Implementation Overview [Back to Top]

The linker takes all its arguments from the command line.  Internally, it makes two passes.  The first reads the input and creates an intermediate representation.  The second finalizes the intermediate representation after all module locations and symbol definitions are known, then outputs a report, a load file, and a symbol table.

6.1                      First Pass [Back to Top]

The first pass parses each record of the input file one at a time, creating in-memory records of type Intermediate, as well as preparing the final logical record of type RecordErrors (so named because it represents both the record and any associated errors).  H and L records are parsed at this point and inserted into the symbol table.  E records are checked to ensure that the counts of records of various types all match.  T and M records are partially parsed (to check program name, etc.), but are primarily handled in the second pass.

6.2                      Second Pass [Back to Top]

The second pass loops over the Intermediate objects created by the first pass.  In the second pass T and M records are fully parsed to generate the final output.  Most errors associated with T and M records are caught in this pass.  After processing via the runSecondPass method, the symbol table, load file, and user report can be created by calling outputLoadMap, outputLoadFile, and outputRecords respectively.

 

7     Simulator Implementation Overview [Back to Top]

The simulator is implemented using a Simulator instance with one Processor instance to handle executing instructions and one Memory instance to simulate an URBAN machine.  The Simulator repeatedly calls the Processor’s public .executeInstruction() method until a halt flag is set by one of the instructions (typically a HALT instruction).  The Simulator also handles printing metadata, such as the memory dump as well as the start and end of the simulation.

Within the Processor class, .executeInstruction() handles most of the overhead work of executing instructions, and then calls the appropriate public method on the Processor to handle the current instruction.  It first increments the Location Counter (this is safe because all the text records in the load file will increment the LC), then reads the instruction from the Memory instance that the Simulator declared.  After reading, an Instruction object is created for it, which will allow useful information from the instruction (such as the instruction’s order bit, registers, binary representation, etc.) to be accessed later.  If debug mode is turned on, everything known about the instruction will be printed to the screen at this point.  After this, all that remains is to use a switch-case statement on the instruction’s opcode to determine which method on the Processor should be called to actually execute the instruction.

 

8     Errata Report [Back to Top]

·         If the programmers provide an integer that requires more than 32 bits, overflow can occur without an error being reported

·         If an EXT/ENT directive fails, then the subsequent labels will not be useable within the program

·         NEWLC does not require a label

·         Star notation expressions within FM operands are only allowed to have one value added or subtracted.

·         Multiple AEXS commands are allowed within a single program, the first is used as the execution start

·         If a symbol is defined multiple times, the last definition is used within the symbol table

·         Shift/rotate/power are allowed to take on any value up to 4095. For power, the result is given modulo 2^32, for rotate/shift, the operations are performed that number of times.

·         DMP is allowed to take values up to 4095, which are then treated modulo 3

·         ADRC expressions may contain as many terms as desired (up to the bounds of a 32-bit integer)

·         Date in object file input to linker must be formatted with commas between hour/minute and between minute/second – colons confuse the record parser

·         Linker replaces any text record that has too many modification records applied to it into a NOP (as well as converting any syntactically or semantically incorrect text record to a NOP if it can read the address)
